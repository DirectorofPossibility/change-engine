PHASE 2: BRAID QUERY LAYER (Week 1-2)
================================================================================
Goal: Build the unified query interface the new frontend needs.
Sits on top of existing 17 modules — does not replace them.

2.1 — Entity Config System
  [ ] Define EntityTypeConfig schema:
      {
        type: string
        table: string
        displayName: string
        pluralName: string
        fields: { title, subtitle, description, image, contacts }
        cardConfig: { variant, showImage, showBadge, showContact }
        detailConfig: { sections, actions, relatedTypes }
        route: string
        color: string
        icon: string
      }
  [ ] Create config entries for all entity types:
      officials, services, policies, content, organizations,
      events, opportunities, agencies, benefits, foundations,
      candidates, learning-paths, guides, neighborhoods
  [ ] Export typed config map: ENTITY_TYPES: Record<string, EntityTypeConfig>

2.2 — Braid Data Functions
  [ ] buildBraidedFeed(options):
      - Input: { pathway?, focusArea?, zip?, center?, entityTypes?, limit }
      - Output: mixed array of typed entities, sorted by relevance
      - Uses RPC functions from Phase 1.2
  [ ] getEntityProfile(type, id):
      - Input: entity type + id
      - Output: full entity + connections + related entities + actions
      - Wraps getWayfinderContext() + entity-specific data
  [ ] getStreamData(pathway, zip?):
      - Input: pathway (or "all") + optional zip
      - Output: grouped sections for the Stream template
      - Groups: "What's New", focus area clusters, journey steps, officials
  [ ] getMapData(bounds?, layers?, zip?):
      - Input: map bounds + active layers + zip
      - Output: GeoJSON + entity counts per area + braided feed for selection
  [ ] searchAcrossTypes(query, filters?):
      - Input: search term + optional type/pathway/center/geo filters
      - Output: mixed results with type labels and relevance scores

2.3 — Type Safety
  [ ] Define BraidedEntity union type (discriminated by entityType field)
  [ ] Define StreamSection type (hero, focus-group, journey, officials)
  [ ] Define EntityProfile type (entity + connections + related + actions)
  [ ] Zod schemas for runtime validation at system boundaries

PHASE 3: NEW FRONTEND (Week 2-3)
================================================================================
Goal: Build the complete new UI. 5 templates, 1 card, 1 wayfinder.
Developed alongside old frontend — no routes replaced yet.

3.1 — Design Tokens + CSS Foundation
  [ ] Define CSS custom properties:
      --pathway-{slug}: color per pathway
      --center-{slug}: color per center
      --level-{slug}: color per gov level
      --brand-*: all brand tokens
  [ ] Tailwind config: extend with CSS variable references
  [ ] Zero inline styles — everything via Tailwind + CSS variables
  [ ] Typography scale: DM Serif Display headings, DM Sans body
  [ ] Responsive breakpoints: mobile-first, 3 breakpoints

3.2 — EntityCard (the one card)
  [ ] Build EntityCard component:
      - Props: entity (BraidedEntity), variant (compact|standard|featured)
      - Image-first: photo/logo/fallback always visible
      - Type badge (pill with entity type color)
      - Pathway dot (small colored indicator)
      - Title, subtitle, description (truncated per variant)
      - Contact actions (email, phone, web — per type config)
      - Click → detail page (Link) or callback (onSelect)

  [ ] Test with all entity types, verify image fallbacks

3.3 — Wayfinder (the one wayfinder)

  [ ] Server component, reads context from props (no client state for initial render)
  [ ] Client interactivity only for: ZIP input, search, filter toggles
  [ ] PathwayDots sub-component: 7 CSS dots, active state, ~40 lines
  [ ] Replace all 10 existing wayfinder components

3.4 — The Stream (homepage + pathway pages)

3.5 — The Profile (all detail pages)

PHASE 4: MIDDLEWARE + ROUTING (Week 3-4)
================================================================================
Goal: Wire personalization, switch routes, progressive rollout.

4.1 — Middleware Personalization
  [ ] Read zip, lang, archetype cookies at the edge
  [ ] Resolve ZIP → neighborhood + district (cached edge lookup)
  [ ] Set request headers: x-neighborhood, x-district, x-lang, x-archetype
  [ ] Server components read headers via headers() — no client context needed
  [ ] NeighborhoodContext becomes optional (only for interactive ZIP changes)

4.2 — Route Structure
  [ ] New route map:
      /                          → Stream (homepage)
      /pathway/[slug]            → Stream (pathway-filtered)
      /explore/[entityType]      → Explorer (any entity listing)
      /[entityType]/[id]         → Profile (any entity detail)
      /map                       → Map
      /compass                   → Compass
      /search                    → Explorer with search active
      /my-area                   → Stream personalized to ZIP
      /elections                 → Custom (keep standalone)
      /dashboard/*               → Keep as-is
      /about, /privacy, /terms   → Static pages (keep)
  [ ] Total: ~15 routes (down from 99)
  [ ] Set up redirects from all old routes to new

4.3 — Caching Strategy
  [ ] Static pages (about, privacy, terms) → full SSG
  [ ] Semi-static (entity listings, pathway streams) → ISR 10min revalidate
  [ ] Dynamic (detail pages, search, my-area) → per-request SSR with stale-while-revalidate
  [ ] Middleware sets Cache-Control by route pattern

4.4 — Progressive Rollout
  [ ] Middleware A/B flag: new-frontend cookie or header
  [ ] Route % of traffic to new templates
  [ ] Monitor Core Web Vitals (LCP, CLS, INP)
  [ ] Roll forward route by route
  [ ] Kill old routes + components once stable
